# Plugin Command: rules-update

You are executing the waves plugin rules update command. Follow these instructions exactly.

## Your Role

You are the main orchestrator for rules updates. Detect code changes and compare them against existing rules to propose additions, modifications, and deprecations. Use specialized agents via Task tool for analysis.

## Step -1: Prerequisites Check (CRITICAL)

1. Check if `ai_files/user_pref.json` exists.
   - IF NOT EXISTS → Display: "⚠️ Run project-init first." → EXIT COMMAND

2. Read `ai_files/user_pref.json`:
   - Extract `user_profile.preferred_language` → Use for all interactions
   - Extract `project_context.project_type` → Determines flow

3. Check rules file exists:
   - Software → `ai_files/project_rules.json`
   - General → `ai_files/project_standards.json`
   - IF NOT EXISTS → Display: "⚠️ No rules found. Run rules-create first." → EXIT COMMAND

**From this point, conduct ALL interactions in the user's preferred language.**

## Step 0: Command Explanation

Display:
```
📘 Command: rules-update

I'll detect code changes and compare them against your existing
rules to find new patterns, necessary modifications, or
deprecated practices.

📁 Rules file: [rules_path]
📊 Current rules: [count by section]

Continue? (Yes/No)
```

IF No → Exit.

## Step 1: Fork by Project Type

- IF `project_type === "software"` → Go to **Flow A or B** (depending on git availability)
- IF `project_type === "general"` → Go to **Flow C**

---

## Flow A: Software (Git-based)

### Step A1: Detect Changed Files via Task Tool

1. Check `.git` exists. If not → go to Flow B.
2. Read existing rules and find the oldest `updated_at` or `created_at` timestamp.
3. Using Task tool, invoke **git-history-analyzer** agent to get file changes since that date.
4. Using Task tool, invoke **change-analyzer** agent to filter autogenerated files.
5. Filter to code files only (exclude docs, configs unless they affect rules).

Display progress:
```
🔍 Detecting changes since [date]...

📝 Scanning git history...
📁 [count] files changed

Filtering autogenerated files...
```

### Step A2: Map Changes to Layers

Using the manifest's `architecture_patterns_by_layer`:
- Map each changed file to its architectural layer
- Only re-analyze layers that have changed files
- Display which layers are affected:

```
📊 Affected layers:
  • [Layer 1] - [count] files changed
  • [Layer 2] - [count] files changed
```

### Step A3: Re-analyze Affected Layers via Task Tool

For each affected layer, launch IN PARALLEL using Task tool:
- **pattern-extractor** — scoped to changed files in this layer
- **convention-detector** — scoped to changed files in this layer
- **antipattern-detector** — scoped to changed files in this layer

Display progress:
```
🔍 Analyzing affected layers:

[✅] Layer 1: Patterns detected
[⏳] Layer 2: Analyzing...
```

### Step A4: Compare with Existing Rules via Task Tool

Using Task tool, invoke **rule-comparator** agent to:
- Compare existing rules against newly detected patterns/conventions
- Identify completely new patterns not in current rules
- Identify rules that may need modification based on changes
- Identify rules that may be outdated/deprecated

### Step A5: Present Proposals

Display:
```
📋 Rules update proposals:

═══════════════════════════════════════
➕ NEW RULES ([count])
═══════════════════════════════════════
[For each: section, proposed description, evidence]

═══════════════════════════════════════
✏️ MODIFICATIONS ([count])
═══════════════════════════════════════
[For each: rule ID, current vs proposed, evidence]

═══════════════════════════════════════
⚠️ DEPRECATIONS ([count])
═══════════════════════════════════════
[For each: rule ID, current rule, evidence for removal]

═══════════════════════════════════════
🔴 NEW ANTIPATTERNS ([count])
═══════════════════════════════════════
[For each: location, problem, suggestion]

Options:
1. Apply all proposals
2. Review each one
3. Cancel
```

### Step A6: Apply Updates

For approved changes:
1. Add new rules with next available IDs.
2. Update modified rules (set `updated_at`).
3. Remove deprecated rules.
4. Validate against schema.

### Step A7: Success Message

```
✅ Rules updated!

📁 File: ai_files/project_rules.json

📊 Changes applied:
  • New rules: [count]
  • Modified: [count]
  • Deprecated: [count]

⚠️ Antipatterns found: [count]
(Review for code quality improvements)

🎯 Continue working with updated rules.
```

## Flow B: Software (Timestamp-based)

### Step B1: Find Changed Files via Task Tool

1. Check `.git` does NOT exist.
2. Read existing rules and find the oldest `updated_at` or `created_at` timestamp.
3. Using Task tool, invoke **change-analyzer** agent to detect files modified since that date.
4. Filter to code files only.

Display:
```
📋 Changed files detected (timestamp-based):
  • Modified: [count]
  • New: [count]

⚠️ Note: Without Git, deleted files cannot be detected.
```

### Step B2: Filter Autogenerated Files

Using Task tool, invoke agent to filter autogenerated files from the list.

### Step B3: Re-analyze Affected Layers via Task Tool

Same as Flow A Step A3 — analyze changed code with pattern/convention/antipattern detectors.

### Step B4: Compare and Present Proposals

Same as Flow A Steps A4-A5 — compare rules and present proposals.

### Step B5: Apply Updates

For approved changes, apply same as Flow A Step A6.

Display additional note:
```
⚠️ Note: Without Git, deleted files cannot be detected.

If you deleted code that was represented in the rules,
you may need to manually deprecate those rules.
```

Success message:
```
✅ Rules updated!

📁 File: ai_files/project_rules.json

📊 Changes applied:
  • New rules: [count]
  • Modified: [count]
  • Deprecated: [count]
```
